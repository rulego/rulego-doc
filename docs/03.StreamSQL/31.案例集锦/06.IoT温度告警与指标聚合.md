# IoT 温度告警与指标聚合案例

## 案例概述

物联网温度监控场景里，往往同时存在两类需求：一是**单点越限即时告警**（每来一条读数就要判断并下发），二是**按设备滚动聚合指标**（每分钟统计均值/极值，用于仪表盘与异常检测）。两类需求对应 StreamSQL 的两种工作模式：非聚合的同步转换（`EmitSync`）与基于窗口的异步聚合（`Emit` + `AddSink`）。

本案例围绕一个统一的温度读数流 `stream`，串联四个子场景：

| 子场景 | 模式 | 窗口 | 核心能力 |
|--------|------|------|----------|
| A. 单点温度告警 | 非聚合（同步） | 无 | `EmitSync` + `CASE WHEN` 即时转换/过滤 |
| B. 滚动 1 分钟设备指标 | 事件时间聚合 | `TumblingWindow('1m')` | Watermark、乱序容忍、空闲推进 |
| C. 高基数设备异常检测 | 处理时间聚合 | `CountingWindow(50)` | `STATETTL` 死 key 清理 |
| D. 确定性测试 | 触发器 | `TumblingWindow('5s')` | `TriggerWindow()` 立即冲窗口 |

## 场景 A：单点温度告警（非聚合，EmitSync）

### 业务目标

每条温度读数到达后**立即**完成：华氏度换算、分级标注（OK/WARNING/CRITICAL），并只下发温度大于 30 的读数。要求一进一出、同步可阻塞调用，调用方直接拿到结果。

### SQL

```sql
SELECT deviceId,
       temperature,
       temperature * 1.8 + 32 AS temp_f,
       CASE WHEN temperature > 35 THEN 'CRITICAL'
            WHEN temperature > 30 THEN 'WARNING'
            ELSE 'OK' END AS level
FROM stream
WHERE temperature > 30
```

### 完整可运行示例

```go
package main

import (
	"fmt"

	"github.com/rulego/streamsql"
)

// 场景 A：非聚合模式，调用 EmitSync 同步拿到单条转换/过滤结果。
// 注意：EmitSync 只能用于非聚合查询；聚合查询必须用 Emit + AddSink。
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId,
			       temperature,
			       temperature * 1.8 + 32 AS temp_f,
			       CASE WHEN temperature > 35 THEN 'CRITICAL'
			            WHEN temperature > 30 THEN 'WARNING'
			            ELSE 'OK' END AS level
			FROM stream
			WHERE temperature > 30`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	// 4 条读数：只有 3 条满足 temperature > 30，会下发；28 那条被过滤。
	readings := []map[string]interface{}{
		{"deviceId": "dev-01", "temperature": 28.0}, // 被过滤
		{"deviceId": "dev-02", "temperature": 32.0}, // WARNING
		{"deviceId": "dev-03", "temperature": 38.0}, // CRITICAL
		{"deviceId": "dev-04", "temperature": 42.0}, // CRITICAL
	}

	for _, r := range readings {
		result, err := ssql.EmitSync(r)
		if err != nil {
			fmt.Printf("error: %v\n", err)
			continue
		}
		if result == nil {
			// WHERE 过滤掉的行：返回 nil，不是错误
			fmt.Printf("[filtered] deviceId=%v\n", r["deviceId"])
			continue
		}
		fmt.Printf("[alert]   %+v\n", result)
	}
}
```

### 预期输出与行为说明

```text
[filtered] deviceId=dev-01
[alert]   map[deviceId:dev-02 level:WARNING temp_f:89.6 temperature:32]
[alert]   map[deviceId:dev-03 level:CRITICAL temp_f:100.4 temperature:38]
[alert]   map[deviceId:dev-04 level:CRITICAL temp_f:107.6 temperature:42]
```

- `EmitSync` 对**非聚合**查询同步返回单条结果；不命中 `WHERE` 的行返回 `nil`（不是 error），可用于区分"被过滤"与"出错"。
- 对聚合查询调用 `EmitSync` 会返回错误：`synchronous mode only supports non-aggregation queries`。
- `CASE WHEN ... THEN ... ELSE ... END` 既可做分级标注，也可配合 `WHERE` 做条件过滤。

### 调优建议

- 转换 / 清洗 / 阈值告警类需求**优先用 `EmitSync`**：同步语义、调用方直接拿到结果，省去回调与 channel 编排。
- 需要异步下游（写入 MQ、批量落库）时，改用 `Emit` + `AddSink(func([]map[string]interface{}))`，下游按批接收。
- `temp_f` 这类算术字段可直接写表达式并 `AS` 别名，无需预先注册 UDF。

## 场景 B：滚动 1 分钟设备指标（事件时间 + Watermark）

### 业务目标

按设备维度，每 1 分钟输出一次窗口内的样本数、平均/最高/最低温度。要求：

1. 以读数自带的时间戳为准（事件时间），而非处理时间；
2. 容忍网络抖动导致的 5 秒乱序；
3. 设备掉线后窗口仍能最终关闭，不留死数据。

### SQL

```sql
SELECT deviceId,
       COUNT(*) AS samples,
       AVG(temperature) AS avg_t,
       MAX(temperature) AS max_t,
       MIN(temperature) AS min_t,
       window_start()  AS w_start
FROM stream
WHERE temperature IS NOT NULL
GROUP BY deviceId, TumblingWindow('1m')
WITH (TIMESTAMP='ts', TIMEUNIT='ms', MAXOUTOFORDERNESS='5s', IDLETIMEOUT='10s')
```

`WITH` 各参数含义：

| 参数 | 含义 |
|------|------|
| `TIMESTAMP='ts'` | 用字段 `ts` 作为事件时间 |
| `TIMEUNIT='ms'` | `ts` 的单位是毫秒（默认 `s`） |
| `MAXOUTOFORDERNESS='5s'` | Watermark = `max(event_time) - 5s`，容忍 5 秒乱序 |
| `IDLETIMEOUT='10s'` | 数据源空闲 10 秒后，按处理时间推进 Watermark，避免掉线设备卡死窗口 |

### 完整可运行示例

```go
package main

import (
	"fmt"
	"time"

	"github.com/rulego/streamsql"
)

// 场景 B：事件时间聚合。ts 为毫秒。窗口 [0, 60000) 内混合 dev-01 / dev-02 的乱序读数；
// 最后发一条 ts=66000 的 dev-03 读数，把 Watermark 推过 60000，触发第一个窗口输出。
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId,
			       COUNT(*) AS samples,
			       AVG(temperature) AS avg_t,
			       MAX(temperature) AS max_t,
			       MIN(temperature) AS min_t,
			       window_start()  AS w_start
			FROM stream
			WHERE temperature IS NOT NULL
			GROUP BY deviceId, TumblingWindow('1m')
			WITH (TIMESTAMP='ts', TIMEUNIT='ms', MAXOUTOFORDERNESS='5s', IDLETIMEOUT='10s')`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	// 窗口结果以"批"的形式回调：本批可能包含同一窗口内多个设备的聚合结果。
	ssql.AddSink(func(batch []map[string]interface{}) {
		for _, row := range batch {
			fmt.Printf("[window] %+v\n", row)
		}
	})

	// 全部落入第一个 1 分钟窗口 [0, 60000ms)，部分故意乱序（在 5s 容忍内）
	readings := []map[string]interface{}{
		{"deviceId": "dev-01", "ts": int64(1000), "temperature": 31.0},
		{"deviceId": "dev-01", "ts": int64(3000), "temperature": 33.0},
		{"deviceId": "dev-02", "ts": int64(2000), "temperature": 29.0}, // 相对 dev-01@3000 乱序
		{"deviceId": "dev-01", "ts": int64(1500), "temperature": 30.0}, // 乱序 1.5s，在 5s 内
		{"deviceId": "dev-02", "ts": int64(4000), "temperature": 27.0},
	}
	for _, r := range readings {
		ssql.Emit(r)
	}

	// 推进 Watermark：ts=66000 落入下一窗口，使 Watermark = 66000 - 5000 = 61000 >= 60000，
	// 第一个窗口 [0,60000) 关闭并输出 dev-01 / dev-02 各自的聚合。
	ssql.Emit(map[string]interface{}{
		"deviceId": "dev-03", "ts": int64(66000), "temperature": 25.0,
	})

	time.Sleep(500 * time.Millisecond) // 等异步 sink 落盘
}
```

### 预期输出与行为说明

```text
[window] map[avg_t:31.333... deviceId:dev-01 max_t:33 min_t:30 samples:3 w_start:0]
[window] map[avg_t:28 deviceId:dev-02 max_t:29 min_t:27 samples:2 w_start:0]
```

- 输出**不是逐条**，而是窗口关闭时按批回调；同一窗口内每个 `deviceId` 一行。
- `dev-01@1500` 晚于 `dev-01@3000` 到达，但因 `MAXOUTOFORDERNESS='5s'` 仍被正确纳入窗口。
- 若没有任何后续数据，`IDLETIMEOUT='10s'` 会在静默 10 秒后用处理时间推进 Watermark，关闭窗口。
- `window_start()` 返回**纳秒**时间戳；如需毫秒展示请自行除以 `1e6`。

### 调优建议

- 周期性报表、监控大盘这类"按时段出点"的需求，**用 `TumblingWindow`**；滑动平滑用 `SlidingWindow('30s','10s')`。
- 事件时间会带来延迟（必须等 Watermark 越过 `window_end` 才出结果）。`MAXOUTOFORDERNESS` 越大越稳但越迟，按业务可接受的乱序幅度调。
- 处理时间模式延迟最低（窗口到点立即关），适合对乱序不敏感的纯时序数据——此时省去 `TIMESTAMP`/`TIMEUNIT` 即可。

## 场景 C：高基数设备异常检测（CountingWindow + State TTL）

### 业务目标

设备规模很大（数万到数十万），其中一部分设备只上报几次就离线。要求：**每凑满 50 条读数**输出一次该设备的统计（均值 + 标准差），同时不能让"永不凑满 50 条"的设备把状态撑爆内存。

### SQL

```sql
SELECT deviceId,
       COUNT(*)           AS cnt,
       AVG(temperature)   AS avg_t,
       STDDEV(temperature) AS stddev_t
FROM stream
GROUP BY deviceId, CountingWindow(50)
WITH (STATETTL='1h')
```

要点：

- `CountingWindow(50)` 按**条数**触发，与时间无关，**只支持处理时间**（不解析 `TIMESTAMP`/`TIMEUNIT`）。
- `STATETTL='1h'`：超过 1 小时未再次触发的 group key 会被后台清理（lazy，`Start` goroutine 中执行）。
- **默认 `STATETTL=0` 表示禁用**（与 Flink 对齐）。也就是说，不开 `STATETTL` 时，只上报过几条的设备会把计数器状态一直留在内存里——这就是"高基数死 key"问题。

### 完整可运行示例

```go
package main

import (
	"fmt"
	"time"

	"github.com/rulego/streamsql"
)

// 场景 C：CountingWindow(50)。dev-A 上报 50 条凑满触发一次；dev-B 只上报 3 条就消失，
// 不开 STATETTL 会泄漏状态，这里用 WITH(STATETTL='1h') 兜底清理。
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId,
			       COUNT(*)            AS cnt,
			       AVG(temperature)    AS avg_t,
			       STDDEV(temperature) AS stddev_t
			FROM stream
			GROUP BY deviceId, CountingWindow(50)
			WITH (STATETTL='1h')`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	ssql.AddSink(func(batch []map[string]interface{}) {
		for _, row := range batch {
			fmt.Printf("[burst] %+v\n", row)
		}
	})

	// dev-A 上报 50 条，温度在 20.0 ~ 24.0 间规律波动，凑满即触发
	for i := 0; i < 50; i++ {
		ssql.Emit(map[string]interface{}{
			"deviceId":   "dev-A",
			"temperature": 20.0 + float64(i%5),
		})
	}

	// dev-B 只上报 3 条就消失——永不触发，STATETTL 负责回收
	for i := 0; i < 3; i++ {
		ssql.Emit(map[string]interface{}{
			"deviceId":   "dev-B",
			"temperature": 99.0,
		})
	}

	time.Sleep(300 * time.Millisecond)
}
```

### 预期输出与行为说明

```text
[burst] map[avg_t:22 deviceId:dev-A cnt:50 stddev_t:1.414...]
```

- `dev-A` 凑满 50 条立即输出，`cnt=50`，`avg_t=22.0`（20..24 均值），`stddev_t≈1.414`（标准差，样本波动）。
- `dev-B` 永远不会出现在输出里，但它的状态会保留：**没有 `STATETTL` 时，大量这样的设备会把内存撑爆**；开启 `STATETTL='1h'` 后，超过 1 小时未再触发的 key 被清理。
- `STDDEV` 为总体标准差；如需样本标准差用 `STDDEVS`。

### 调优建议

- 固定批量触发（每 N 条出一个点）的场景**用 `CountingWindow`**：与时间无关，吞吐稳定时延迟可预期。
- 设备 / 用户 / 会话 ID 维度数极大时，**务必开 `STATETTL`**——它就是为高基数 key 准备的兜底防线。
- `STATETTL` 仅对 `CountingWindow` 生效；时间窗口靠 `IDLETIMEOUT` 推进 Watermark 自 closing，不需要 `STATETTL`。
- 不要给 `CountingWindow` 配 `TIMESTAMP`/`TIMEUNIT`：它是纯处理时间语义。

## 场景 D（加餐）：用 TriggerWindow 写确定性测试

### 业务目标

时间窗口在测试里很难用——你不想真的 `time.Sleep(5*time.Second)` 等窗口关闭。`TriggerWindow()` 提供了一个手动触发当前窗口立即输出的能力，让测试既快又确定。

### 完整可运行示例

```go
package main

import (
	"fmt"
	"time"

	"github.com/rulego/streamsql"
)

// 场景 D：TumblingWindow('5s')，但只 Emit 一条，然后用 TriggerWindow() 立即冲窗口，
// 不必等满 5 秒。这正是 StreamSQL 自带测试套件避免长 sleep 的做法。
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId, COUNT(*) AS cnt
			FROM stream
			GROUP BY deviceId, TumblingWindow('5s')`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	ssql.AddSink(func(batch []map[string]interface{}) {
		for _, row := range batch {
			fmt.Printf("[triggered] %+v\n", row)
		}
	})

	ssql.Emit(map[string]interface{}{"deviceId": "dev-X"})
	time.Sleep(200 * time.Millisecond) // 让数据进入窗口

	// 不等 5 秒，手动触发当前窗口立即输出
	ssql.TriggerWindow()

	time.Sleep(200 * time.Millisecond) // 让 sink 落盘
}
```

### 预期输出与行为说明

```text
[triggered] map[cnt:1 deviceId:dev-X]
```

- `TriggerWindow()` 对**时间窗口**立即冲窗口并输出；对 `CountingWindow` 是 no-op（按条数触发，与时间无关）；对非窗口查询也是安全 no-op。
- 因此它可以放心写进测试 teardown：无论被测 SQL 是哪种形态，都不会 panic。

### 调优建议

- 写 StreamSQL 的端到端测试时，**用 `TriggerWindow()` 取代 `time.Sleep(windowSize)`**：测试从秒级降到毫秒级，且不依赖调度抖动。
- 生产侧极少用——只在"明确知道现在要结算"的业务节点（如收到关机信号、批次封口）才有意义。

## 总结与最佳实践

**按数据特征选窗口：**

- 周期性报表 / 监控大盘 → `TumblingWindow` / `SlidingWindow`（时间驱动）
- 固定批量结算 → `CountingWindow`（条数驱动）
- 用户会话 / 设备在线段 → `SessionWindow('5m')`（gap 驱动）

**按延迟要求选时间语义：**

- 处理时间：最低延迟，窗口到点即关；不要 `TIMESTAMP`/`TIMEUNIT`
- 事件时间 + Watermark：准确但有延迟，配 `MAXOUTOFORDERNESS` 容忍乱序、`IDLETIMEOUT` 防掉线卡死

**按返回要求选 API：**

- 单点转换 / 过滤 / 阈值告警 → `EmitSync`（非聚合，同步拿单条结果）
- 窗口聚合 / 异步下游 → `Emit` + `AddSink`（批回调）/ `ToChannel()`（channel）

**高基数防内存泄漏：**

- `CountingWindow` 上线前先评估 group key 基数；高基数务必开 `STATETTL`（默认 `0` 禁用，与 Flink 对齐）
- 时间窗口不必开 `STATETTL`，靠 `IDLETIMEOUT` 推进 Watermark 即可关闭窗口
