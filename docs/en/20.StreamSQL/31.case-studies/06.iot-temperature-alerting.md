# IoT Temperature Alerting and Metrics Aggregation

## Case Overview

IoT temperature monitoring usually carries two kinds of requirements at once: **per-reading alerting** (every reading must be evaluated and dispatched the moment it arrives) and **rolling per-device metrics** (avg / max / count over a window, feeding dashboards and anomaly detection). These map directly onto StreamSQL's two operating modes: a non-aggregation synchronous transform (`EmitSync`) and a windowed asynchronous aggregation (`Emit` + `AddSink`).

This case study is built around a single temperature reading stream `stream` and walks through four sub-scenarios:

| Scenario | Mode | Window | Key Capability |
|----------|------|--------|----------------|
| A. Per-reading alerting | Non-aggregation (sync) | none | `EmitSync` + `CASE WHEN` transform/filter |
| B. Rolling 1-minute device metrics | Event-time aggregation | `TumblingWindow('1m')` | Watermark, out-of-order tolerance, idle advance |
| C. High-cardinality anomaly detection | Processing-time aggregation | `CountingWindow(50)` | `STATETTL` dead-key reaping |
| D. Deterministic test | Manual trigger | `TumblingWindow('5s')` | `TriggerWindow()` immediate flush |

## Scenario A: Per-Reading Alerting (Non-Aggregation, EmitSync)

### Business Goal

For every temperature reading, **immediately** compute the Fahrenheit value, classify it (OK / WARNING / CRITICAL), and dispatch only readings above 30 degrees. The call must be synchronous and blocking, returning the result to the caller.

### SQL

```sql
SELECT deviceId,
       temperature,
       temperature * 1.8 + 32 AS temp_f,
       CASE WHEN temperature > 35 THEN 'CRITICAL'
            WHEN temperature > 30 THEN 'WARNING'
            ELSE 'OK' END AS level
FROM stream
WHERE temperature > 30
```

### Complete Runnable Example

```go
package main

import (
	"fmt"

	"github.com/rulego/streamsql"
)

// Scenario A: non-aggregation mode. EmitSync returns the single transformed
// (or filtered) row synchronously. Note: EmitSync only works for non-aggregation
// queries; aggregation queries must use Emit + AddSink.
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId,
			       temperature,
			       temperature * 1.8 + 32 AS temp_f,
			       CASE WHEN temperature > 35 THEN 'CRITICAL'
			            WHEN temperature > 30 THEN 'WARNING'
			            ELSE 'OK' END AS level
			FROM stream
			WHERE temperature > 30`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	// 4 readings: only 3 satisfy temperature > 30 and are dispatched; 28 is filtered out.
	readings := []map[string]interface{}{
		{"deviceId": "dev-01", "temperature": 28.0}, // filtered
		{"deviceId": "dev-02", "temperature": 32.0}, // WARNING
		{"deviceId": "dev-03", "temperature": 38.0}, // CRITICAL
		{"deviceId": "dev-04", "temperature": 42.0}, // CRITICAL
	}

	for _, r := range readings {
		result, err := ssql.EmitSync(r)
		if err != nil {
			fmt.Printf("error: %v\n", err)
			continue
		}
		if result == nil {
			// Rows that fail the WHERE clause return nil — this is not an error.
			fmt.Printf("[filtered] deviceId=%v\n", r["deviceId"])
			continue
		}
		fmt.Printf("[alert]   %+v\n", result)
	}
}
```

### Expected Output and Behavior

```text
[filtered] deviceId=dev-01
[alert]   map[deviceId:dev-02 level:WARNING temp_f:89.6 temperature:32]
[alert]   map[deviceId:dev-03 level:CRITICAL temp_f:100.4 temperature:38]
[alert]   map[deviceId:dev-04 level:CRITICAL temp_f:107.6 temperature:42]
```

- `EmitSync` synchronously returns a single result row for **non-aggregation** queries. A row that fails `WHERE` returns `nil` (not an error), so you can distinguish "filtered out" from "error".
- Calling `EmitSync` on an aggregation query returns an error: `synchronous mode only supports non-aggregation queries`.
- `CASE WHEN ... THEN ... ELSE ... END` works both for classification columns and, combined with `WHERE`, for conditional filtering.

### Tuning Tips

- For transform / cleansing / threshold-alert workloads, **prefer `EmitSync`**: synchronous semantics, caller gets the result directly, no callback or channel plumbing.
- When you need an async downstream (writing to a message queue, batch persistence), switch to `Emit` + `AddSink(func([]map[string]interface{}))` and receive results in batches.
- Arithmetic columns like `temp_f` can be written inline as expressions and aliased with `AS` — no need to register a UDF.

## Scenario B: Rolling 1-Minute Device Metrics (Event Time + Watermark)

### Business Goal

Per device, emit once per 1-minute window: sample count, average, max, and min temperature. Requirements:

1. Use the timestamp carried by the reading (event time), not processing time.
2. Tolerate up to 5 seconds of out-of-order arrival caused by network jitter.
3. Still close the window even when a device goes offline — do not leak dead windows.

### SQL

```sql
SELECT deviceId,
       COUNT(*) AS samples,
       AVG(temperature) AS avg_t,
       MAX(temperature) AS max_t,
       MIN(temperature) AS min_t,
       window_start()  AS w_start
FROM stream
WHERE temperature IS NOT NULL
GROUP BY deviceId, TumblingWindow('1m')
WITH (TIMESTAMP='ts', TIMEUNIT='ms', MAXOUTOFORDERNESS='5s', IDLETIMEOUT='10s')
```

Meaning of each `WITH` option:

| Option | Meaning |
|--------|---------|
| `TIMESTAMP='ts'` | Use field `ts` as the event time |
| `TIMEUNIT='ms'` | `ts` is in milliseconds (default `s`) |
| `MAXOUTOFORDERNESS='5s'` | Watermark = `max(event_time) - 5s`; tolerate 5s of out-of-order data |
| `IDLETIMEOUT='10s'` | When the source is idle for 10s, advance the watermark by processing time so an offline device does not stall the window |

### Complete Runnable Example

```go
package main

import (
	"fmt"
	"time"

	"github.com/rulego/streamsql"
)

// Scenario B: event-time aggregation. ts is in milliseconds. The first window
// [0, 60000) mixes out-of-order readings from dev-01 / dev-02; a final dev-03
// reading at ts=66000 pushes the watermark past 60000 and closes the window.
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId,
			       COUNT(*) AS samples,
			       AVG(temperature) AS avg_t,
			       MAX(temperature) AS max_t,
			       MIN(temperature) AS min_t,
			       window_start()  AS w_start
			FROM stream
			WHERE temperature IS NOT NULL
			GROUP BY deviceId, TumblingWindow('1m')
			WITH (TIMESTAMP='ts', TIMEUNIT='ms', MAXOUTOFORDERNESS='5s', IDLETIMEOUT='10s')`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	// Window results are delivered as a batch: one batch may carry the aggregated
	// rows of multiple devices from the same window.
	ssql.AddSink(func(batch []map[string]interface{}) {
		for _, row := range batch {
			fmt.Printf("[window] %+v\n", row)
		}
	})

	// All readings land in the first 1-minute window [0, 60000ms); some are
	// intentionally out of order but within the 5s tolerance.
	readings := []map[string]interface{}{
		{"deviceId": "dev-01", "ts": int64(1000), "temperature": 31.0},
		{"deviceId": "dev-01", "ts": int64(3000), "temperature": 33.0},
		{"deviceId": "dev-02", "ts": int64(2000), "temperature": 29.0}, // out-of-order vs dev-01@3000
		{"deviceId": "dev-01", "ts": int64(1500), "temperature": 30.0}, // 1.5s late, within 5s
		{"deviceId": "dev-02", "ts": int64(4000), "temperature": 27.0},
	}
	for _, r := range readings {
		ssql.Emit(r)
	}

	// Advance the watermark: ts=66000 falls into the next window, so Watermark
	// = 66000 - 5000 = 61000 >= 60000, which closes window [0,60000) and emits
	// the per-device aggregates for dev-01 and dev-02.
	ssql.Emit(map[string]interface{}{
		"deviceId": "dev-03", "ts": int64(66000), "temperature": 25.0,
	})

	time.Sleep(500 * time.Millisecond) // let the async sink drain
}
```

### Expected Output and Behavior

```text
[window] map[avg_t:31.333... deviceId:dev-01 max_t:33 min_t:30 samples:3 w_start:0]
[window] map[avg_t:28 deviceId:dev-02 max_t:29 min_t:27 samples:2 w_start:0]
```

- Output is **not per-row** — it is delivered as a batch when the window closes, with one row per `deviceId` in that window.
- `dev-01@1500` arrives after `dev-01@3000` but is still placed in the window correctly thanks to `MAXOUTOFORDERNESS='5s'`.
- If no further data arrives, `IDLETIMEOUT='10s'` advances the watermark by processing time after 10s of silence and closes the window.
- `window_start()` returns a **nanosecond** timestamp; divide by `1e6` if you need to display it in milliseconds.

### Tuning Tips

- For periodic reporting and dashboard metrics, **use `TumblingWindow`**; use `SlidingWindow('30s','10s')` for smoothed rolling stats.
- Event time introduces latency — a window fires only after the watermark passes `window_end`. Larger `MAXOUTOFORDERNESS` is more robust but later; tune it to the actual arrival skew of your pipeline.
- Processing-time mode has the lowest latency (the window closes on the dot) and is fine for time-series data that does not care about ordering — just omit `TIMESTAMP`/`TIMEUNIT`.

## Scenario C: High-Cardinality Anomaly Detection (CountingWindow + State TTL)

### Business Goal

The fleet is large (tens of thousands to hundreds of thousands of devices), and a non-trivial fraction report a few times and then go silent. Requirement: **every 50 readings**, emit per-device stats (mean + standard deviation), and do not let "never-quite-reaches-50" devices blow up memory.

### SQL

```sql
SELECT deviceId,
       COUNT(*)            AS cnt,
       AVG(temperature)    AS avg_t,
       STDDEV(temperature) AS stddev_t
FROM stream
GROUP BY deviceId, CountingWindow(50)
WITH (STATETTL='1h')
```

Key points:

- `CountingWindow(50)` is triggered by **count**, not time, and supports **processing time only** (`TIMESTAMP`/`TIMEUNIT` are ignored).
- `STATETTL='1h'`: group keys that have not triggered again for over 1 hour are reaped by a background goroutine (lazy, started from `Start`).
- **The default `STATETTL=0` is disabled** (Flink-aligned). Without `STATETTL`, a device that reports only a few times keeps its counter state in memory forever — this is the high-cardinality dead-key problem.

### Complete Runnable Example

```go
package main

import (
	"fmt"
	"time"

	"github.com/rulego/streamsql"
)

// Scenario C: CountingWindow(50). dev-A reports 50 times and fires once;
// dev-B reports 3 times and disappears — it would leak state without STATETTL,
// so we enable WITH(STATETTL='1h') to reap the dead key.
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId,
			       COUNT(*)            AS cnt,
			       AVG(temperature)    AS avg_t,
			       STDDEV(temperature) AS stddev_t
			FROM stream
			GROUP BY deviceId, CountingWindow(50)
			WITH (STATETTL='1h')`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	ssql.AddSink(func(batch []map[string]interface{}) {
		for _, row := range batch {
			fmt.Printf("[burst] %+v\n", row)
		}
	})

	// dev-A reports 50 times, temperature cycles 20.0..24.0; the window fires on the 50th.
	for i := 0; i < 50; i++ {
		ssql.Emit(map[string]interface{}{
			"deviceId":   "dev-A",
			"temperature": 20.0 + float64(i%5),
		})
	}

	// dev-B reports 3 times and vanishes — never fires; STATETTL reaps it.
	for i := 0; i < 3; i++ {
		ssql.Emit(map[string]interface{}{
			"deviceId":   "dev-B",
			"temperature": 99.0,
		})
	}

	time.Sleep(300 * time.Millisecond)
}
```

### Expected Output and Behavior

```text
[burst] map[avg_t:22 deviceId:dev-A cnt:50 stddev_t:1.414...]
```

- `dev-A` fires the moment the 50th reading arrives: `cnt=50`, `avg_t=22.0` (mean of 20..24), `stddev_t≈1.414` (population standard deviation).
- `dev-B` never appears in the output, but its state would otherwise be retained: **without `STATETTL`, many such devices inflate memory**; with `STATETTL='1h'`, any key inactive for over 1 hour is reaped.
- `STDDEV` is population standard deviation; use `STDDEVS` for sample standard deviation.

### Tuning Tips

- For fixed-volume batching (emit one point per N readings), **use `CountingWindow`**: it is time-independent and its latency is predictable when throughput is stable.
- When the group key cardinality (devices / users / sessions) is very large, **always enable `STATETTL`** — it is the dedicated safety net for high-cardinality keys.
- `STATETTL` applies only to `CountingWindow`; time windows close themselves via `IDLETIMEOUT`-driven watermark advance and do not need `STATETTL`.
- Do not pair `CountingWindow` with `TIMESTAMP`/`TIMEUNIT` — it is strictly processing-time.

## Scenario D (Bonus): Deterministic Tests with TriggerWindow

### Business Goal

Time windows are awkward in tests — you do not want to literally `time.Sleep(5*time.Second)` waiting for a window to close. `TriggerWindow()` lets you manually trigger the current window to emit immediately, making tests fast and deterministic.

### Complete Runnable Example

```go
package main

import (
	"fmt"
	"time"

	"github.com/rulego/streamsql"
)

// Scenario D: TumblingWindow('5s'), but we Emit only one row and then call
// TriggerWindow() to flush immediately rather than waiting 5 seconds. This is
// exactly how the StreamSQL test suite avoids long sleeps.
func main() {
	ssql := streamsql.New()
	defer ssql.Stop()

	sql := `SELECT deviceId, COUNT(*) AS cnt
			FROM stream
			GROUP BY deviceId, TumblingWindow('5s')`

	if err := ssql.Execute(sql); err != nil {
		panic(err)
	}

	ssql.AddSink(func(batch []map[string]interface{}) {
		for _, row := range batch {
			fmt.Printf("[triggered] %+v\n", row)
		}
	})

	ssql.Emit(map[string]interface{}{"deviceId": "dev-X"})
	time.Sleep(200 * time.Millisecond) // let the row enter the window

	// Don't wait 5s — manually flush the current window.
	ssql.TriggerWindow()

	time.Sleep(200 * time.Millisecond) // let the sink drain
}
```

### Expected Output and Behavior

```text
[triggered] map[cnt:1 deviceId:dev-X]
```

- `TriggerWindow()` immediately flushes the current **time window**. For `CountingWindow` it is a no-op (count-driven, time-independent); for non-window queries it is also a safe no-op.
- That makes it safe to call in a test teardown regardless of which SQL shape is under test — it will never panic.

### Tuning Tips

- When writing StreamSQL end-to-end tests, **replace `time.Sleep(windowSize)` with `TriggerWindow()`**: tests drop from seconds to milliseconds and become immune to scheduling jitter.
- It is rarely used in production — only at business points where you genuinely know "settle now" (a shutdown signal, a batch seal-off).

## Summary and Best Practices

**Pick the window by data shape:**

- Periodic reporting / dashboards → `TumblingWindow` / `SlidingWindow` (time-driven)
- Fixed-volume settlement → `CountingWindow` (count-driven)
- User sessions / device online segments → `SessionWindow('5m')` (gap-driven)

**Pick the time semantics by latency need:**

- Processing time: lowest latency, fires on the dot; omit `TIMESTAMP`/`TIMEUNIT`
- Event time + watermark: accurate but latent; pair `MAXOUTOFORDERNESS` (out-of-order tolerance) with `IDLETIMEOUT` (anti-stall)

**Pick the API by return semantics:**

- Per-row transform / filter / threshold alert → `EmitSync` (non-aggregation, synchronous single-row result)
- Windowed aggregation / async downstream → `Emit` + `AddSink` (batch callback) / `ToChannel()` (channel)

**Guard memory under high cardinality:**

- Before shipping a `CountingWindow`, estimate group-key cardinality; enable `STATETTL` for high cardinality (default `0` is disabled, Flink-aligned)
- Time windows do not need `STATETTL` — `IDLETIMEOUT` advances the watermark and closes them
